# My_PID_Motor 项目复盘存档

这是一份项目过程复盘，用来记录项目做了什么、为什么这样做、过程中遇到过什么问题、哪些判断来自实测，以及当前结论的边界。它不是正式简历，也不是对外宣传稿。

## 1. 项目目标

项目最初的目标，是基于 STM32F103 完成一个可实际运行的直流电机控制系统，逐步接入 PWM 输出、编码器测速、ADC 输入、PID 控制和 UART 命令交互。后续又将原有控制结构迁移到 FreeRTOS，通过任务拆分和命令队列整理执行链路。控制功能基本稳定后，项目重点转向真实现象诊断和参数验证，最终通过实测确定当前默认 Kp，并形成一套以后能够回顾的代码、release 和验证文档。

## 2. 当前实现内容

当前工程已经具备 PWM 电机输出、Encoder 速度反馈、ADC/UART 目标源、正反向控制、启动助推、前馈和 PID 修正。UART 支持 `run=1`、`run=0`、`stop`、`status`、`get fault`、`t=xxx`、`kp/ki/kd`、`set target uart`、`set target adc`、`rst` 等命令，PID 参数可以在线设置。

RTOS 层包含 `ControlTask`、`CmdTask` 和 `LogTask`，控制命令通过 `CmdQueue` 传递，串口文本发送由 UART TX mutex 保护。工程还包含 fault 状态和故障快照查询接口、`v2.0-rtos-cmdqueue` 发布基线、`Kp=0.05` 调参记录、`target=600` 负载扰动结果以及用于展示当前状态的 README。

## 3. 系统结构回顾

```text
UART RX
  ↓
CmdTask / App_Cmd_Parse
  ↓
CmdQueue
  ↓
ControlTask（10 ms）
  ↓
PID + Feedforward
  ↓
PWM / Motor Driver → Motor
                       ↓
               Encoder Feedback
                       ↓
                  ControlTask

LogTask → 周期 status 输出
```

`CmdTask` 负责读取和解析串口命令，查询类命令直接响应，控制类命令投递到队列。`ControlTask` 以 10 ms 周期统一消费命令、更新控制状态、读取速度反馈并计算电机输出。`LogTask` 负责周期状态输出。UART TX mutex 不是控制算法的一部分，它解决的是多个任务同时输出时可能发生的文本冲突。

## 4. 从裸机到 RTOS 的变化

早期结构中，控制、串口和日志逻辑容易相互牵连。迁移后，这三类职责分别交给 `ControlTask`、`CmdTask` 和 `LogTask`：周期控制不再等待串口输入，命令处理不直接承担控制循环，日志也有独立节奏。后续再加入 `CmdQueue`，避免串口任务直接修改控制状态。这个改造没有改变项目性质，但使结构更清楚，任务职责也更容易核对。

## 5. CmdQueue 解决的问题

串口接收具有突发性，而控制执行要求相对稳定的周期，两者不适合混在同一条执行路径中。引入 `CmdQueue` 后，解析任务只负责识别和投递命令，控制任务成为修改控制状态的统一入口，从而降低多任务同时操作控制状态造成混乱的风险，也为队列报错和处理延迟提供了明确观察点。

项目已完成 20 条和 50 条连续命令压力验证：在对应测试条件下，没有观察到 ERR、队列满、丢失入队回应或测试后系统卡死。100 条压力测试已经暂停，没有执行，也不能写成通过。

## 6. PID 与前馈调参过程

### 6.1 零 PID 前馈基线

调参没有直接从增加 PID 参数开始，而是先在 `Kp=0, Ki=0, Kd=0` 条件下观察前馈本身。正反向的 `±500、±600、±800、±1000` 档位均能持续运行，方向正确，记录中为 `fault:NONE`。这个步骤把“前馈能否提供基本输出”和“PID 是否在修正误差”分开，避免把机械下限、助推或前馈问题误认为 PID 参数问题。

### 6.2 Kp=0.05 验证

随后先在 `target=±600` 条件下验证 `Kp=0.05`，再扩展到 `±500、±600、±800、±1000` 八个档位。整理后的八档平均误差不超过约 11 RPM，正反向表现基本对称，PWM 只在前馈值附近小幅修正，记录中未观察到明显振荡，全程为 `fault:NONE`。

因此当前默认参数确定为：

```c
#define APP_PID_DEFAULT_KP 0.05f
#define APP_PID_DEFAULT_KI 0.0f
#define APP_PID_DEFAULT_KD 0.0f
```

没有继续加入 Ki/Kd，主要是因为当前稳态误差已经小于或接近约 42.9 RPM 的测速量化粒度，同时没有明显振荡或大过冲现象。现有记录不足以证明增加 Ki/Kd 会带来明确收益，反而可能增加积分累积、低速抖动和调试复杂度。

## 7. 负载扰动测试结论

测试条件为 UART 目标源、`target=600`、`Kp=0.05, Ki=0, Kd=0`，负载由人工轻微施加。

| 阶段 | actual 均值 | 最低 actual | PWM 均值 | 最高 PWM |
| --- | ---: | ---: | ---: | ---: |
| 扰动前 | 604.7 | — | 127.5 | — |
| 第一次负载 | 580.7 | 471 | 128.7 | 134 |
| 第一次释放 | 594.7 | — | 128.1 | — |
| 第二次负载 | 568.8 | 385 | 129.3 | 138 |
| 第二次释放 | 601.5 | — | 127.7 | — |

两次扰动期间均观察到 actual 下降、PWM 同步提高；释放后 actual 回到 600 附近，PWM 回到约 128 的基线附近。全程 `fault:NONE`，最终 stop 后 `enable:0、State:IDLE、PWM:0、actual:0`。因此，这次测试只支持定性判断：`Kp=0.05` 下具备基本轻载扰动补偿能力。

人工负载的大小和动作时刻没有精确测量，这份记录不能用于计算精确恢复时间、控制带宽或严格动态性能，也不能表述为高精度动态控制验证。

## 8. 验证记录与证据链

项目目前保留了以下主要记录：`release/v2.0-rtos-cmdqueue/` 对应 RTOS 命令队列基线；`cmd_queue_pressure_test.md` 记录 20/50 条命令压力；`fault_snapshot_review.md` 记录快照字段评审；`control_tuning_summary_kp_0p05.md` 记录前馈与 Kp 调参；`load_disturbance_600_result.md` 记录轻载扰动结果；`verification_record_audit.md` 核对证据等级；README 用于说明当前项目状态。

证据链并非所有部分都同等完整。一些早期原始日志未归档进仓库；复位后显示 `kp:0.050` 属于用户人工确认，没有单独完整日志；负载扰动日志能够支持变化趋势，却不能支持精确动态指标；真实 fault snapshot 的有快照分支也未经过实际故障验证。现有文档尽量区分了实测结果、人工确认、代码评审和暂停项，不用后来的文字包装代替缺失证据。

## 9. 踩过的坑和经验

这个项目后半段最明显的经验，是测试数量不等于诊断价值。`stop/status` 很重要，但它们更适合作为安全前置和收尾检查，不需要单独包装成主要成果。命令压力做到 20/50 条、没有出现 ERR 和队列满后，继续堆到 100 条并不能自动带来更有价值的判断，所以及时暂停。

控制参数也不能只改宏而不保留依据。默认 Kp 最终能够从 0 改为 0.05，是因为前馈基线、正反向全范围和轻载扰动记录能够相互对应。相反，测试期间为提高观察密度使用的 `APP_LOG_TICK=200` 不属于正式配置，结束后恢复为 1000。人工扰动还提醒了一点：能看见趋势，不等于测出了带宽和恢复时间。文档必须明确哪些是实测、哪些是人工确认、哪些只是代码判断或已暂停项目。

## 10. 当前限制

当前主要验证路径是 UART target；ADC target source 已经实现，但没有作为本阶段主展示路径。Ki/Kd 当前未启用，也没有进行破坏性故障注入或严格的控制带宽、恢复时间、频响测试。负载扰动只是人工轻载条件，部分原始日志仍保存在项目目录外。项目应按本科阶段嵌入式控制学习与实践工程理解，不是工业产品；这些内容是结论适用范围，而不是需要用文字掩盖的缺点。

## 11. 后续是否继续维护

My_PID_Motor 在当前阶段可以暂停。若未来重新维护，更值得优先考虑的是结构化改进或工具化，例如日志解析脚本、上位机可视化、参数持久化或 ADC target 低速策略统一，而不是继续堆叠相似测试。这些只是未来可能讨论的方向，不是当前阶段的待办，也不要求现在继续实施。

## 12. 未来可提炼的点

以后回顾这个项目，可以从 STM32F103 外设综合应用、FreeRTOS 多任务拆分、串口命令队列设计、PID 参数实测调参、电机闭环控制诊断、验证边界意识，以及从功能实现到项目文档化的完整过程几个方向提炼。真正值得保留的不是“做过多少测试”，而是逐步把控制现象、代码结构和证据记录对应起来的过程。
