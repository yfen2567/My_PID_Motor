# My_PID_Motor v2.1 诊断验证计划

> 本计划是 v2.1 诊断验证计划，不代表测试已经全部完成。

## 1. v2.1 目标

`v2.1-diagnostic-verification` 的目标是为当前固件建立可追溯的诊断验证证据链，包括计划、原始日志、结果报告、异常记录、构建产物信息和版本标识。本阶段只做诊断验证封版，不增加控制功能，也不以扩大测试数量代替问题定位。

## 2. 已有 v2.0 基础

仓库已有 STM32F103、PWM、Encoder、ADC、PID、UART 和 FreeRTOS 实现，应用任务包括 `ControlTask`、`CmdTask`、`LogTask`。控制命令通过 `CmdQueue` 从解析任务传递到控制任务，UART 文本发送由 mutex 保护。

`v2.0-rtos-cmdqueue` 已保留 release 基线。已有文档还记录了基础 run/stop/status、非法及越界命令、20/50 条命令压力、无故障快照查询、`Kp=0.05` 全范围调参和一组 `target=600` 人工轻载扰动结果。这些记录可作为 v2.1 的参考基线，但不能自动代替本计划要求的新证据。

## 3. 本轮计划验证内容

1. 按低、中、较高目标速度执行三组负载扰动测试，保存原始串口日志和分组报告；
2. 对照 `docs/verification/log_field_dictionary.md` 检查 status 的字段名、顺序、格式和取值完整性；
3. 检查 `get fault` 无快照返回，并在设备自然出现 fault 时核对 fault snapshot；本轮不主动制造破坏性故障；
4. 使用 LLCOM 脚本或等价方式进行一次可复现的半自动串口回归；
5. 选择少量代表性非法命令和越界命令，确认错误返回明确且控制状态未被意外改变；
6. 检查 `run=1` 后的 `enable/State/PWM`，以及 `stop` 后 `enable:0`、`State:IDLE`、`PWM:0` 的软件状态；
7. 对所有测试记录固件 commit、构建产物标识、串口配置、测试时间和硬件条件。

## 4. 本轮不验证内容

- FOC 或新的伺服控制算法；
- 洗袜机功能及其他与 My_PID_Motor 无关的功能；
- OTA、无线调试、云平台；
- 复杂 GUI 或复杂上位机；
- 破坏性故障注入、长期可靠性或工业认证；
- 未经单独确认的控制算法、任务结构或硬件初始化变更。

## 5. 测试环境记录

每次测试开始前填写以下信息，缺失项标记“未记录”，不能事后编造。

| 项目 | 本轮记录要求 |
| --- | --- |
| MCU / 工程 | STM32F103C8Tx / My_PID_Motor |
| 固件版本 | `v2.1-diagnostic-verification` 候选版本及 Git commit hash |
| 构建产物 | ELF/HEX/BIN 文件名、生成时间、文件大小和 SHA-256 |
| 电机与驱动 | 实际型号、供电电压、连接方式；未知则写“未记录” |
| 反馈 | TIM2 Encoder；每圈计数配置以当前源码为准 |
| 目标源 | 负载测试统一使用 UART target |
| PID 默认值 | `Kp=0.05, Ki=0, Kd=0`；测试前由 status 核对 |
| 串口 | USART1；波特率等参数按当前工程和串口工具实际配置记录 |
| 行结束符 | 真实 CRLF，不发送字面量 `\r\n` |
| 日志时间 | 记录本地时间、固件 `ms` 和测试阶段标记 |

## 6. 测试工具

- STM32CubeIDE：仅用于后续获准的候选固件构建和产物确认；
- LLCOM 或等价串口工具：发送命令、记录 TX/RX 和阶段 MARK；
- `scripts/`：后续保存半自动回归或日志解析脚本，本轮只建立目录；
- SHA-256 工具：标识构建产物和需要纳入证据链的原始日志；
- 示波器或逻辑分析仪：仅在确有需要且实际使用时记录波形；未测不得写成已确认。

## 7. 测试数据保存位置

| 数据 | 保存位置 | 要求 |
| --- | --- | --- |
| 原始串口日志 | `logs/` | 保存未改写的 TX/RX、时间戳和 MARK；建议使用 `.txt` |
| 分项测试报告 | `reports/` | 引用对应原始日志、commit 和产物 hash |
| 计划与字段定义 | `docs/verification/` | 与当前源码和输出格式保持一致 |
| 半自动脚本 | `scripts/` | 脚本需记录版本、参数和使用方法 |
| release 证据 | 后续候选 release 目录 | 只在全部必需项确认后整理，本轮不创建 release |

## 8. 通过与失败判据

### 通过

- 测试按记录的命令、硬件和固件条件可复现；
- status 字段完整、格式稳定，关键状态变化与命令及实际现象相符；
- 合法控制命令有明确响应，非法或越界命令被拒绝且不改变控制状态；
- 负载扰动时 actual 与 PWM 呈现可解释的相对变化，释放后回到扰动前附近；
- 全程没有无法解释的 fault、复位、静默丢回应或串口文本破坏；
- stop 后软件状态回到安全停止状态；
- 原始日志、测试报告、commit 和构建产物能够相互对应。

### 失败或需暂停判断

- 命令无响应、回应数量异常、队列报错或测试后 status 无法恢复；
- 字段缺失、格式漂移、数据与当前状态明显矛盾；
- actual 下降时 PWM 没有可解释变化，或释放轻载后无法恢复；
- 出现 fault、异常声音、明显发热、剧烈抖动或失控；
- stop 后 `enable/State/PWM` 未回到预期安全状态；
- 日志、固件或 commit 无法建立对应关系。

人工负载大小不可精确一致，因此负载测试以相对趋势和可复现证据为主，不设置虚假的精密动态阈值。

## 9. v2.1 Release 所需证据

在考虑 v2.1 tag 或 release 前，至少应具备：

- 已冻结的本验证计划和日志字段表；
- 三组负载扰动原始日志及对应报告；
- status 字段完整性检查记录；
- 半自动 UART 回归原始输出和统计结果；
- 非法/越界命令及 run/stop 行为记录；
- `get fault` 无快照结果；若没有自然 fault，必须明确有快照分支仍未验证；
- 所有异常、限制和延期项的书面说明；
- 候选版本 commit hash、工作树状态、构建环境和构建产物信息；
- 构建产物及关键证据文件的 SHA-256；
- README 中与实际证据一致的验证边界说明；
- 完成 `docs/verification/release_checklist_v2.1.md`，再由项目负责人决定是否打 tag。

本轮不构建、不烧录、不执行测试，也不创建 tag 或 release。
