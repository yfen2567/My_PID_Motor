# v1 阶段性总结

## 版本定位

`My_PID_Motor` v1 阶段的目标，是把项目从早期的“PID 调速实验”整理成一个具备基本工程结构的 STM32 电机闭环控制节点。

当前版本已经形成以下主流程：

```text
目标速度输入
    -> 前馈 + PID 控制计算
    -> PWM + 方向 GPIO 输出
    -> 电机运行
    -> 编码器测速反馈
    -> 故障检测与串口日志输出
```

相比早期版本，v1 不再只关注“电机能不能转起来”，而是开始关注控制节点应具备的完整性，包括输入采集、执行输出、反馈测速、串口交互、故障保护和现场状态记录。

---

## 已完成内容

### 1. 编码器闭环控制主链路

当前工程已经完成直流电机闭环控制的基本链路：

* 使用 `TIM1_CH1` 输出 PWM；
* 使用方向 GPIO 控制电机正反转和停止；
* 使用 `TIM2` 编码器接口读取 AB 相反馈；
* 使用 `TIM3` 定时中断触发 10 ms 控制周期；
* 控制层周期性读取目标速度、实际速度和编码器增量；
* 根据目标速度与实际速度计算 PWM 输出。

这一阶段的重点是让系统具备稳定的闭环控制框架，而不是只进行单次 PWM 输出测试。

---

### 2. 前馈 + PID + 启动助推控制结构

v1 阶段的控制输出由三部分组成：

```text
启动助推
前馈 PWM
PID 修正量
```

其中，前馈项用于给出维持目标速度所需的大致 PWM，PID 用于修正实际速度与目标速度之间的误差。

当前版本已经明确了一个重要调试结论：

> 从静止启动时，单纯增大 `Kp` 并不是最好的办法；启动阶段需要单独的启动助推逻辑，稳定运行阶段再交给前馈和 PID 修正。

这说明项目已经从“只套 PID 公式”进入到实际电机控制调试阶段。

---

### 3. ADC 目标输入与第 2 路辅助采集

v1 阶段已经把 ADC 从单路输入扩展为 2 路采集输入：

```text
adc1：目标速度输入
adc2：辅助采集输入
```

当前第 1 路 ADC 用于目标速度输入，第 2 路 ADC 暂时作为辅助采集量保留。

第 2 路 ADC 当前不直接参与控制，主要用于：

* 验证系统具备多路采集能力；
* 在串口日志中观察额外模拟量；
* 为后续负载、电流、水位、电压或旋钮输入预留接口；
* 为故障快照保存更多现场数据提供基础。

这一步的意义不是马上增加复杂控制算法，而是让项目从“单输入控制 Demo”升级为“多路采集 + 执行输出 + 状态反馈”的控制节点雏形。

---

### 4. 串口命令与在线调试

v1 阶段已经具备基础串口调试能力。

当前串口支持的主要功能包括：

```text
run=1             启动电机控制
stop / run=0      停止电机
t=xxx             设置目标速度
set target adc    使用 ADC 作为目标速度来源
set target uart   使用串口作为目标速度来源
kp=xxx            设置 Kp
ki=xxx            设置 Ki
kd=xxx            设置 Kd
rst               复位 PID 和故障状态
status            输出当前系统状态
help              输出命令帮助
get fault         读取故障快照
```

串口状态输出已经包含：

```text
时间戳
运行使能
系统状态
目标来源
目标速度
实际速度
编码器增量
PWM 输出
ADC1 输入
ADC2 输入
故障状态
PID 参数
```

这使得系统具备了在线观察、在线调参和基本问题排查能力。

---

### 5. 故障保护与故障快照

当前版本已经引入基础故障保护逻辑，主要包括：

```text
超速保护
编码器丢失保护
PWM 长时间饱和保护
```

当系统进入故障状态时，控制层会停止电机输出，并记录故障发生时的关键现场信息。

故障快照当前设计包含：

```text
快照是否有效
故障发生时间
故障类型
系统状态
目标速度
实际速度
PWM 输出
目标 ADC 输入
辅助 ADC 输入
```

相比只输出一个故障码，故障快照可以帮助后续判断故障发生时系统处于什么状态，例如目标速度是否过高、PWM 是否已经接近饱和、ADC 输入是否异常等。

## 当前工程结构

v1 阶段已经形成比较清晰的模块分工：

```text
Motor
    负责 PWM 输出、方向 GPIO 控制和电机停止。

Encoder
    负责读取编码器计数、计算速度和保存编码器增量。

ADC_Sensor
    负责 ADC DMA 采样、滤波、目标速度映射和辅助 ADC 采集。

PID
    负责 PID 参数、积分限幅、输出限幅和 PID 计算。

Control
    负责组合各底层模块，维护系统状态、目标速度、实际速度、PWM 输出和故障状态。

Uart
    负责串口接收、命令解析、状态输出和故障快照查询。
```

当前项目已经具备嵌入式控制工程中常见的分层结构：

```text
底层外设驱动
    -> 应用模块封装
    -> 控制层状态管理
    -> 串口调试接口
    -> 文档与调试记录
```

## v1 阶段成果

v1 阶段已经完成以下核心目标：

* 完成 STM32F103C8Tx 平台上的直流电机闭环控制；
* 完成 PWM 输出、方向控制、编码器测速和 10 ms 周期控制；
* 完成 ADC 目标速度输入；
* 增加第 2 路 ADC 辅助采集输入；
* 完成前馈 + PID + 启动助推的控制结构；
* 完成串口命令解析和在线调试接口；
* 完成基础故障保护；
* 增加故障快照结构；
* 初步形成控制节点工程结构。

可以将当前版本概括为：

```text
2 路采集输入
+ 1 路 PWM 执行输出
+ 编码器速度反馈
+ 串口通信
+ PID 闭环控制
+ 故障保护与快照记录
```

## 当前仍需完善的问题

v1 当前属于阶段性版本，还不是最终稳定版本。后续需要继续完善以下内容：

### 1. 文档与代码同步

当前 README、串口命令说明和源码状态需要进一步统一，尤其是：

* `adc`、`adc1`、`adc2` 的命名；
* 第 2 路 ADC 的用途说明；
* `get fault` 命令说明；
* v1 阶段功能列表。

---

### 2. 故障快照字段补全

当前故障快照已经设计了 `adc_aux` 字段，后续需要确认故障发生时辅助 ADC 值是否被完整保存。

建议故障快照至少包含：

```text
fault
state
target_speed
actual_speed
pwm
adc_target
adc_aux
tick_ms
```

---

### 3. 第二路 ADC 的工程含义需要固定

当前第 2 路 ADC 还只是辅助采集输入。后续可以明确为以下方向之一：

```text
负载模拟输入
电流采样输入
电池电压采样输入
水位/压力/温度等传感器输入
第二路旋钮输入
```

v1 阶段建议先定义为：

```text
辅助采集输入 / 负载模拟输入
```

暂时只进入日志和故障快照，不参与 PID 控制。

---

### 4. 控制参数调试流程需要固化

后续应继续整理调参流程，明确：

```text
先调启动助推
再调前馈参数
最后调 PID 参数
```

避免直接通过增大 PID 参数解决所有问题。

---

## 后续计划

v1 之后可以进入 v2 阶段，重点不再是简单增加功能，而是提高系统稳定性和可维护性。

建议 v2 方向包括：

* 增加速度斜坡，避免目标速度突变；
* 完善故障快照保存和清除逻辑；
* 增加参数保存功能，将 PID 和前馈参数写入 Flash；
* 统一串口通信格式；
* 增加更规范的状态机；
* 增加上位机通信协议；
* 完善文档中的接线说明、调试步骤和故障排查流程。

## 总结

`My_PID_Motor` v1 阶段已经完成了一个基础但相对完整的 STM32 编码器闭环电机控制节点。

当前项目不再只是简单的 PID 调速实验，而是已经具备：

```text
目标输入
控制计算
PWM 执行
编码器反馈
串口调试
故障保护
状态记录
```

这些控制系统的基本组成部分。

后续的重点应放在稳定性、文档同步、故障诊断、参数保存和通信协议规范化上，而不是继续无序增加功能。
