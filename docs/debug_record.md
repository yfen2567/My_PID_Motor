# 调试记录精简版

本文从 `docs/调试记录.md`、`docs/PID调参记录.md`、`docs/问题排查记录.md` 和当前源码整理而来，只记录项目中真实遇到并定位过的问题。

## 1. 串口有状态输出，但 `run=1` / `help` 没反应

**问题**

串口能周期性输出状态，但发送 `run=1`、`help` 等命令没有立即响应。

**原因分析**

周期状态输出来自 `main while` 中的 `Uart_PrintfStatus()`，只能说明 USART1 TX 正常。命令接收走的是另一条链路：`HAL_UART_RxCpltCallback()` 按字节接收，只有收到 `\r` 或 `\n` 后才会置位 `s_cmd_ready`，再由 `Uart_Task()` 解析。

**解决方案**

串口助手发送命令时追加换行，建议使用 CRLF，例如 `run=1\r\n`、`help\r\n`。

**最终结果**

开启 CRLF 后，`help` 能返回命令列表，`run=1` 能返回 `OK:RUN`，确认 USART1 RX、命令结束符和 `Uart_Task()` 链路正常。

## 2. 电机能转，但 `actual/delta` 一直为 0

**问题**

发送较小目标速度后，电机可以转动，但状态中 `actual=0`、`delta=0`，同时 PWM 持续增大。

**原因分析**

电机驱动链路已经工作，但编码器反馈链路没有有效数据。出现过接线问题，导致 TIM2 编码器计数没有随电机转动变化。

**解决方案**

检查编码器 A/B 相、电源、地线和 TIM2 输入引脚连接。调试时用固定目标和较低 PID 参数观察 `delta` 是否随转动变化。

**最终结果**

修复接线后，`actual` 和 `delta` 开始变化，闭环控制具备有效反馈。

## 3. 编码器 A/B 方向接反

**问题**

修复编码器接线后，出现正向 PWM 下 `actual` 为负的情况，闭环输出容易迅速打满或反向。

**原因分析**

TIM2 编码器模式根据 A/B 相相位判断方向。A/B 相接反时，软件读到的速度方向和电机实际驱动方向相反，PID 会把误差越修越大。

**解决方案**

交换编码器 A/B 相接线，或在软件中统一对 `Encoder_GetDelta()` / `Encoder_Getspeed()` 结果取反。调试标准是：正向 PWM 时，`actual` 应主要为正。

**最终结果**

调整 A/B 相后，编码器方向与电机方向一致，后续 PID 调参数据才有意义。

## 4. PA15/PB3 未上拉导致双相编码器异常

**问题**

A/B 两相都接入后，一启动就可能进入 `SPEED_OVER_LIMIT` 或 `PWM_SATURATION`；拔掉其中一路反而现象改善。

**原因分析**

TIM2_CH1 使用 PA15，TIM2_CH2 使用 PB3。两路输入未稳定上拉时，编码器信号容易浮空或受电机 PWM 干扰，双相解码会产生异常计数。故障后的 `status` 只显示停机后的 `actual=0/PWM=0`，不代表触发瞬间没有异常速度。

**解决方案**

在 CubeMX 中将 PA15/PB3 配置为 `GPIO_PULLUP`，保持 TIM2 编码器模式为 `TIM_ENCODERMODE_TI12`。必要时可外接上拉电阻并优化走线。

**最终结果**

PA15/PB3 上拉后，两相同时接入可以正常运行，启动即超速的问题消失。

## 5. 低分辨率编码器导致测速量化明显

**问题**

低速或中低速时，`actual` 容易在 `0/214/428...` 等离散值之间跳变，看起来像速度突然变为 0 或突然升高。

**原因分析**

当前按输出轴一圈约 28 count 计算速度。10 ms 窗口下，`delta=1` 已经约等于 214 RPM；当一个周期内刚好没有计数时，速度会显示为 0。这是低分辨率编码器和短测速窗口共同造成的量化现象。

**解决方案**

在 `Encoder_Getspeed()` 中使用滑动窗口累计多个周期的 `delta`，当前配置为 `APP_ENCODER_SPEED_AVG_N = 5`，等效约 50 ms 测速窗口。调参时同时观察 `actual` 和 `delta`，不要只看单次速度值。

**最终结果**

速度显示比单个 10 ms 窗口更平滑，但响应会变慢。后续调参以窗口后的速度和实际电机现象一起判断。

## 6. 正向目标下出现负 PWM，导致反向制动和震动

**问题**

当 `actual` 略高于 `target` 后，PID 输出可能变成负 PWM，电机表现为一卡一卡或被反向拉住。

**原因分析**

`Motor_SetPWM()` 将有符号 PWM 解释为方向控制：正值正转，负值反转。在单向调速测试中，负 PWM 不是简单“减小驱动力”，而是反向制动。低分辨率测速下 `actual` 的跳变会放大这种现象。

**解决方案**

调试时曾用“正向目标下禁止负 PWM”的方式验证问题。后续设计应区分单向调速和双向调速：单向场景可限制反向 PWM，双向场景则需要按目标方向设计方向切换和 PID reset。

**最终结果**

确认负 PWM 会导致反向拉扯，不再把所有卡顿都简单归因于 Kp。当前项目保留有符号 PWM 能力，后续双向控制需要继续完善方向切换策略。

## 7. Kp 过大导致振荡或超速

**问题**

ADC 目标约 `551~552 RPM` 时，真实测试出现：

- `kp=0.40`：`PWM≈220`，电机不动或启动不足；
- `kp=0.50`：出现卡顿，曾记录到 `actual=685`、`PWM=-67`；
- `kp=0.70`：启动后进入 `SPEED_OVER_LIMIT`。

**原因分析**

纯 P 控制同时承担“启动电机”和“稳定调速”两个任务，容易冲突。Kp 太小无法克服静摩擦，Kp 稍大又可能因启动冲击、速度量化和负 PWM 反拉进入振荡或超速。

**解决方案**

调参时先固定目标来源为 UART，`ki=0`、`kd=0`，只调 Kp。确认编码器方向、上拉和测速窗口后，再引入前馈、启动助推和少量 Ki。

**最终结果**

确认这不是程序突然异常，而是参数、测速量化和电机死区共同造成的控制问题。Kp 不应单独承担全部启动力。

## 8. PWM 多数为正但电机仍抖动

**问题**

在 `kp≈0.47`、ADC 目标约 `551~552 RPM` 时，PWM 多数时间为正，但电机仍然抖动；偶尔出现很小的负 PWM，例如 `PWM=-2`。

**原因分析**

这不是单一原因。接近目标时，P 输出会变小，`PWM=58` 这类值可能落在电机死区附近，无法稳定推动电机；速度掉下去后误差变大，PWM 又突然升高。低分辨率测速还会让 `actual` 偶尔越过目标，导致输出跨过 0。

**解决方案**

把问题拆开处理：负 PWM 反拉和 PWM 太小顶不动不是同一个问题。前者需要方向策略，后者需要前馈、死区补偿或启动助推。

**最终结果**

确认“卡顿”不一定都是负 PWM 造成，也可能是输出落入电机死区。后续控制策略改为前馈 + PID 修正，而不是只靠 Kp。

## 9. 前馈 PWM 能维持转动，但不能可靠从静止启动

**问题**

开环测试用 `(actual=1328, PWM=220)` 和 `(actual=1242, PWM=210)` 估算出 `kS≈66`、`kV≈0.12`。按该前馈计算，`target=551` 时 `PWM_ff≈132`，但 `PWM=132/150/170` 从静止时仍可能不转；手动给电机一点外力后，同样 PWM 又能维持转动。

**原因分析**

`PWM_ff = kS + kV * target` 描述的是已经转起来后的维持 PWM，不等于从静止打破静摩擦所需的启动 PWM。电机启动还受静摩擦、齿轮啮合位置、转子位置和电源瞬态影响。

**解决方案**

增加启动助推状态：`run=1` 时置位 `s_pwm_start_boost_active` 并清零 tick；启动阶段短时间输出较大的 `APP_PWM_START_BOOST`；检测到 `encoder_delta != 0` 或助推超时后退出助推，切回正常 `PWM_ff + PID`。

**最终结果**

启动可靠性提高，控制策略被拆成三部分：启动助推负责打破静摩擦，前馈负责提供大致维持力，PID 负责误差修正。

## 10. ADC 目标源影响 PID 调参判断

**问题**

使用 ADC 作为目标速度源时，`target` 会随电位器和 ADC 滤波结果轻微变化，例如记录中常见 `target=551~552`。调参时容易把目标变化和 PID 参数效果混在一起。

**原因分析**

当前 `Control_Update()` 每周期都会执行 `ADC_Update()`。当 `s_adc_target_enable` 有效时，会通过 `ADC_MapTargetSpeed()` 周期性刷新 `target_speed`。这适合作为电位器目标输入，但不适合做严格的 PID 参数对比实验。

**解决方案**

PID 调参时先发送 `set target uart`，再使用 `t=<num>` 固定目标速度。ADC 输入继续用于正常电位器控制场景。

**最终结果**

固定 UART 目标后，调参数据更容易对比；ADC 目标源保留为可用输入方式。

## 当前调试结论

- 串口命令必须带 `\r` 或 `\n`，推荐 CRLF。
- 编码器调试优先级高于 PID 调参；没有正确方向和稳定上拉时，PID 数据没有参考价值。
- 当前编码器分辨率较低，速度显示存在量化，调参时要结合 `delta` 和实际电机表现。
- Kp 不能同时承担启动和稳定控制；当前项目已引入前馈和启动助推来分担这两件事。
- 故障后的状态行可能已经被停机逻辑清零，不能完全代表故障触发瞬间的数据。
