# My_PID_Motor v2.0 诊断验证总结

## 1. 本轮范围

- 固件版本：`My_PID_Motor v2.0-rtos-cmdqueue`（由测试人员记录）。
- 测试日期：2026-06-27。
- 本轮只汇总串口命令队列压力和 fault snapshot 字段评审，不修改源码，不追加 100 条压力测试或故障注入。
- 结论依据：当前代码阅读、20/50 条 LLCOM 原始日志以及真实 `get fault` 输出。

## 2. 已确认

| 项目 | 真实证据 | 已确认结论 |
|---|---|---|
| 20 条命令压力 | `D:\llcom_uart_pressure_log.txt`；真实 CRLF；20 条 `kp=0.01` 对应 20 个 `OK:CMD_QUEUED` | 未见 ERR、队列满、分配失败、丢失入队回应或协议级粘连；结束后 status 正常，`kp:0.010`，周期输出继续 |
| 50 条命令压力 | `D:\llcom_uart_pressure_50_log.txt`；真实 CRLF；配置 `kpDelay=20 ms`；50 条发送及索引标记对应 50 个 `OK:CMD_QUEUED` | 未见 ERR、队列满、分配失败、丢失入队回应或协议级粘连；结束后 status 正常，`kp:0.010`，周期输出继续 |
| 无快照查询 | 2026-06-27 14:53:07 实际发送 `get fault` | 固件明确返回 `FAULT_SNAPSHOT_NONE`，随后 status 正常，`fault:NONE`；无历史快照分支通过 |
| 当前快照字段 | 代码中的 `FaultSnapshot_t` 与输出格式 | 已有 `valid`、`time`、`state`、`fault`、`T_speed`、`A_speed`、`pwm`、`adc_target`、`adc_aux`，足够对首次故障做粗分类 |

在已测试条件下，20 条与 50 条压力测试均通过，未发现命令入口/入队报错或系统卡死。当前证据不要求立即修改代码。

## 3. 未确认

- 100 条或更高速率下的极限队列容量与恢复行为：未测试。
- 50 条测试配置为 20 ms 间隔，但日志只有秒级时间戳，实际逐条毫秒间隔：未确认。
- `OK:CMD_QUEUED` 只证明成功入队。由于压力命令均为相同的 `kp=0.01`，每一条是否都被 ControlTask 分别执行：无法由最终 `kp:0.010` 单独确认。
- 有真实快照时的输出完整性、字段值、采集时机及根因定位效果：未确认。
- `adc_aux` 虽存在于快照结构和输出中，但当前保存代码未赋值，不能视为有效测量。
- 物理 PWM、方向引脚、编码器原始计数和电机机械行为不在本轮串口证据范围内。
- 当前日志记录了固件版本名，但未绑定唯一源码提交或固件文件哈希；日志与唯一构建产物的对应关系：未确认。

## 4. 后续触发条件

| 触发情况 | 后续动作 |
|---|---|
| 需要验证队列极限、实际使用速率高于本轮条件，或出现丢回应/队列报错 | 再执行 100 条或更高速率测试，并保留完整原始日志；否则不做 |
| 设备自然进入 fault | 立即保存故障前后 status、`get fault` 原始输出和当时操作条件，不先复位 |
| 真实快照字段不足以定位故障，或业务确实需要可信 `adc_aux` | 先提交最小代码变更方案，获得确认后再修改和验证 |
| 出现 `actual=0`、编码器异常、PWM 有输出但电机不动、PID 振荡或 ADC 抖动 | 转入对应的专项诊断，采集串口日志并按需要补充波形或测量记录 |

## 5. 阶段结论

本轮两个最小诊断任务已经完成：命令队列在已测试的 20/50 条条件下表现稳定；fault snapshot 的无快照分支可用，现有字段在代码层面够做首次故障粗分类，但真实有快照分支仍待自然故障证据。暂不执行 100 条测试，不进行故障注入，不修改源码。
