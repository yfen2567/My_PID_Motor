# fault snapshot 最小字段评审

## 目的

当前只评审已有字段是否足够做首次故障的粗定位，不安排复杂故障注入。

## 当前字段

| 字段 | 能回答什么 |
|---|---|
| `valid` | 是否存在历史快照 |
| `time` | 首次故障相对开机的时间 |
| `state` | 保存快照时的软件状态；当前代码实际更接近故障前状态 |
| `fault` | 首次捕获的故障类型 |
| `T_speed` | 故障时的目标速度 |
| `A_speed` | 故障时最近一次计算速度 |
| `pwm` | 故障检查前的软件 PWM 命令 |
| `adc_target` | 主 ADC 原始值 |
| `adc_aux` | 当前代码未赋值，不能作为有效测量 |

无快照时，`get fault` 返回 `FAULT_SNAPSHOT_NONE`。

## 缺少的关键字段

| 缺失字段 | 导致无法判断的问题 |
|---|---|
| `enable`、明确的故障前/后 state | 保护动作是否完整执行 |
| `source` | target 来自 ADC 还是 UART |
| `encoder_delta` | actual=0 是否与编码器脉冲一致 |
| 方向引脚、TIM compare | 软件 pwm 是否对应真实输出 |
| 最近执行命令 | 故障是否紧跟 run/stop/参数修改 |
| 有效 `adc_aux` | 辅助 ADC 在故障时是否异常 |

## 现在最小是否够用

够做粗分类：可以看到首次 fault、相对时间、故障前状态、target、actual、软件 PWM 和主 ADC。

不够做根因确认：无法证明物理 PWM、编码器原始反馈、命令顺序和辅助 ADC 是否正常。

本次真实串口测试只确认了“无历史快照”分支能够给出明确结果；有快照分支的字段格式、字段值和采集时机尚未通过真实故障日志验证，均为**未确认**。

## 后续是否需要改代码

当前不立即改代码。先等待一次真实 fault snapshot；只有现有字段确实阻止定位时，再提交最小变更方案。

已确认的最小候选是：若继续保留 `adc_aux`，应在获得修改授权后补充真实赋值或明确移除，不能维持“输出字段存在但未采集”的状态。

真实 `get fault` 输出：

```text

[2026/06/27 14:53:07.227]  →  FAULT_SNAPSHOT_NONE␍␊
[2026/06/27 14:53:07.340]  →  ms:33389,enable:0,State:IDLE,source:ADC,target:551,actual:0,delta:0,PWM:0,adc1:2257,adc2:2049,fault:NONE,kp:0.000,ki:0.000,kd:0.000␍␊
```

结论：无历史故障快照时，`get fault` 能明确返回 `FAULT_SNAPSHOT_NONE`，随后 status 继续正常输出；该分支验证通过。有快照分支尚未实测，暂不据此提出源码修改。
