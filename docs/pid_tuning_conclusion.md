# PID 调参结论

本文根据当前 PID 源码、`Control.c` 控制逻辑，以及已有调试记录整理。结论只描述已经在项目中出现过或能从代码确认的内容，不把未验证参数写成最终参数。

## 1. PID 参数设置

当前 PID 参数由两部分决定：

| 参数来源 | 当前情况 |
|---|---|
| 源码默认值 | `APP_PID_DEFAULT_KP = 0.0f`，`APP_PID_DEFAULT_KI = 0.0f`，`APP_PID_DEFAULT_KD = 0.0f` |
| 串口在线设置 | 支持 `kp=<num>`、`ki=<num>`、`kd=<num>`，范围检查为 `0 <= value <= 100` |
| 控制周期 | `APP_CONTROL_DT_SEC = 0.01`，即 10 ms 控制周期 |
| PID 输出限幅 | `APP_PID_OUTPUT_MAX = 1000.0` |

`PID_Update()` 的实际计算逻辑为：

```c
error = target_speed - actual_speed;
integral += error * dt;
derivative = (error - last_error) / dt;

output = kp * error + ki * integral + kd * derivative;
```

同时当前 PID 代码包含：

- 积分限幅：根据 `ki` 和 `output_abs_MAX` 计算积分上限；
- 输出限幅：PID 输出限制在 `±output_abs_MAX`；
- 参数设置后复位：`Control_SetPID()` 调用 `PID_SetGains()` 后会执行 `PID_Reset()`；
- 启停/故障/方向变化时复位：`Control_SetEnable()`、`Control_EnterFault()`、`Control_SetTargetSpeed()` 的方向变化分支都会复位 PID。

注意：当前电机输出不是单纯 `PID_Update()`，而是：

```text
启动助推阶段：PWM = APP_PWM_START_BOOST
正常控制阶段：PWM = Control_CalcFeedForward(target) + PID_Update(target, actual)
```

因此调参时不能只看 PID 参数，还要同时考虑前馈和启动助推。

## 2. 调参顺序

当前项目比较可靠的调参顺序如下：

1. 先确认电机驱动链路正常：PWM 输出、IN1/IN2 方向控制、电源和共地正常。
2. 再确认编码器链路正常：`actual/delta` 能随电机转动变化，且正向 PWM 时 `actual` 主要为正。
3. 确认 PA15/PB3 已上拉。上拉前的数据不能作为 PID 调参依据。
4. 调参时先切到 UART 目标源：发送 `set target uart`，再用 `t=<num>` 固定目标。
5. 初期设置 `ki=0`、`kd=0`，只调 `kp`。
6. 找到不会明显振荡、不触发故障的 Kp 范围。
7. 区分启动问题和稳态问题：启动不足优先用启动助推处理，不要只靠增大 Kp。
8. 前馈和启动助推稳定后，再考虑少量增加 Ki。
9. Kd 最后再评估；当前低分辨率测速下，Kd 对噪声和速度跳变很敏感。

## 3. Kp 对系统影响

Kp 是当前调试中影响最明显的参数。已有真实记录如下：

| 目标速度 | Kp | Ki | Kd | 现象 | 结论 |
|---|---:|---:|---:|---|---|
| `551~552` | `0.40` | `0.00` | `0.00` | `PWM≈220`，`actual=0`，电机不动或启动不足 | Kp 偏小，输出不足以可靠启动 |
| `551~552` | `0.47` | `0.00` | `0.00` | 多数时间 PWM 为正，但仍抖动；曾出现 `PWM=58/159/219` 和很小负 PWM | 接近目标时 P 输出可能落入电机死区 |
| `551` | `0.50` | `0.00` | `0.00` | 记录到 `actual=685`、`PWM=-67`，电机一卡一卡 | 已接近不稳定区，测速跳变和负 PWM 反向制动会放大卡顿 |
| `551` | `0.70` | `0.00` | `0.00` | 启动后进入 `SPEED_OVER_LIMIT` | Kp 偏大，启动冲击过强 |

调试结论：

- Kp 增大后响应会变快，但也更容易过冲、振荡或触发超速故障。
- 当前电机存在静摩擦和死区，Kp 太小会启动不足。
- 不能为了“让电机从静止启动”无限增大 Kp，因为这样稳定运行时容易过冲。
- 当前项目已经引入启动助推和前馈，Kp 应主要负责误差修正，而不是承担全部启动力。

## 4. Ki 对系统影响

从代码逻辑看，Ki 的作用是累积长期误差：

```text
i_term = ki * integral
```

它适合用来消除 P 控制留下的稳态误差。但当前已有调试记录中，没有一组在“编码器方向正确、PA15/PB3 已上拉、测速窗口稳定、启动助推和前馈已加入”之后验证过的 Ki 最终数据。

已知情况：

- 早期出现过 `ki=0.10`，但当时编码器反馈链路和方向还没有完全排除问题，因此不能作为有效调参结论。
- 后续有效排查 Kp、测速量化、负 PWM、静摩擦时，主要使用 `ki=0.00`。

当前结论：

- Ki 暂时不应作为启动手段。
- Ki 应在 P、前馈、启动助推都稳定后再少量加入。
- 加 Ki 后需要观察长期误差是否减小，同时注意积分过大导致恢复慢或再次振荡。

## 5. Kd 对系统影响

从代码逻辑看，Kd 根据误差变化率产生输出：

```text
d_term = kd * derivative
derivative = (error - last_error) / dt
```

理论上，Kd 可以抑制误差快速变化。但当前项目的速度反馈存在明显量化：

- 编码器按输出轴一圈约 `28 count` 计算；
- 10 ms 下 `delta=1` 已约等于 `214 RPM`；
- 当前使用 `APP_ENCODER_SPEED_AVG_N = 5` 的滑动窗口改善显示，但速度仍不是连续平滑量。

已知情况：

- 早期使用过 `kd=0.10`，当时曾出现 PWM 剧烈变化和打满现象；
- 由于当时还叠加了编码器接线、方向和反馈不稳定问题，这组数据不能单独证明 Kd 的最终效果；
- 但从当前测速量化特点看，Kd 对速度跳变会很敏感，容易放大噪声。

当前结论：

- Kd 初期保持 `0.00` 是合理的。
- 等 P 或 PI 控制稳定后，再考虑是否需要 Kd。
- 如果后续要使用 Kd，应先改善速度滤波或引入更稳定的速度估计。

## 6. 当前最终参数

这里的“最终参数”指当前源码已经固化采用的默认参数，不等于已经完成全工况标定的最优控制参数。

当前源码中固化的 PID 默认参数为：

```c
#define APP_PID_DEFAULT_KP 0.0f
#define APP_PID_DEFAULT_KI 0.0f
#define APP_PID_DEFAULT_KD 0.0f
```

这说明当前工程没有把某一组 Kp/Ki/Kd 作为最终固定参数写入代码。运行时参数通过串口命令设置，并通过状态输出查看：

```text
kp=<num>
ki=<num>
kd=<num>
status
```

当前已经写入源码并参与控制的非 PID 参数为：

```c
#define APP_PWM_FF_KS 66.0f
#define APP_PWM_FF_KV 0.12f
#define APP_PWM_START_BOOST 900
#define APP_PWM_START_BOOST_TICK 10
```

这些参数来自后期调试结论：

- `kS/kV` 用于稳定运行前馈；
- `APP_PWM_START_BOOST` 用于打破静摩擦；
- PID 只在正常控制阶段作为误差修正。

因此当前“最终结论”不是某个固定 `kp/ki/kd`，而是控制策略调整为：

```text
启动助推 + 前馈 PWM + PID 修正
```

其中 PID 当前默认从 `0/0/0` 开始，具体 Kp 仍需要根据目标速度、负载和供电通过串口继续标定。

## 7. 调参过程中遇到的问题

### 7.1 编码器反馈缺失

电机能转，但 `actual=0`、`delta=0`，导致 PID 误认为速度一直没有起来，PWM 持续增大。最终定位为编码器反馈链路问题。

### 7.2 编码器方向反

正向 PWM 下 `actual` 为负，导致 PID 计算出的误差越来越大，闭环越调越错。通过交换 A/B 相或软件取反思路解决。

### 7.3 PA15/PB3 未上拉

双相编码器同时接入时启动即超速或饱和，最后确认 PA15/PB3 需要上拉。上拉前的数据不能作为 PID 调参依据。

### 7.4 测速量化明显

低分辨率编码器在短窗口下导致 `actual` 跳变，容易误判为电机速度剧烈变化。当前用滑动窗口改善，但仍需要结合 `delta` 判断。

### 7.5 负 PWM 反向制动

正向目标下，当 `actual > target` 时，PID 可能输出负 PWM。由于 `Motor_SetPWM()` 会把负 PWM 变成反向驱动，这在单向调速中表现为反向制动和震动。

### 7.6 Kp 过大误判为程序问题

`kp=0.40` 启动不足，`kp=0.50` 开始卡顿，`kp=0.70` 触发超速。最终判断主要是 Kp、速度量化、电机死区和负 PWM 共同作用，不是代码主链路突然异常。

### 7.7 前馈不能替代启动助推

用开环数据估算得到的前馈 `PWM_ff≈132` 能描述维持转动，但不能保证从静止启动。原因是静摩擦和启动滞回比稳定运行阻力更大。

### 7.8 ADC 目标源影响调参

ADC 目标源会周期性刷新 `target_speed`，记录中常见 `target=551~552` 的轻微变化。严格调参时应先切到 UART 目标源。

## 8. 为什么最终采用当前参数

当前源码采用 `kp=0.0f, ki=0.0f, kd=0.0f` 作为默认值，主要原因是：

1. 已有调试记录没有证明某一组 Kp/Ki/Kd 已经在不同目标速度下稳定可用。
2. 早期包含 `ki=0.10`、`kd=0.10` 的数据受到编码器接线、方向和反馈不稳定影响，不能作为最终参数。
3. 当前系统最大问题不是单纯 PID 公式，而是电机静摩擦、低分辨率测速、编码器输入稳定性和负 PWM 方向策略。
4. 后期控制策略已经把“启动”和“维持运行”从 PID 中拆出来：
   - 启动由 `APP_PWM_START_BOOST` 负责；
   - 大致维持 PWM 由 `APP_PWM_FF_KS / APP_PWM_FF_KV` 负责；
   - PID 只负责误差修正。
5. 保持默认 PID 为 0，可以避免烧录后直接使用未验证参数导致电机猛转、振荡或触发故障。

当前最合理的使用方式是：

```text
1. 上电后先确认 status 正常。
2. set target uart
3. t=<固定目标>
4. ki=0
5. kd=0
6. 从较小 kp 开始逐步增加。
7. 观察 target / actual / delta / PWM / fault。
8. 前馈和启动助推稳定后，再少量尝试 Ki。
```

一句话总结：

```text
当前项目还没有固化最终 Kp/Ki/Kd；已经固化的是“启动助推 + 前馈 + PID 修正”的控制思路。PID 参数应继续通过串口在真实负载下标定。
```
