# 问题排查记录

问题排查记录用于保存项目调试中遇到的典型问题。以后复盘和面试时，可以用这里的记录说明自己不是只会跑 Demo，而是能定位问题。

它不是普通日记，而是专门记录：

```text
出现过什么问题
怎么观察
怎么判断
怎么处理
最后是否解决
```

## 常见问题排查思路

| 现象 | 优先怀疑 | 排查动作 |
|---|---|---|
| `pwm` 很大但 `actual` 一直为 0 | 电机驱动、供电、PWM 引脚、编码器接线 | 先手动给 PWM，看电机和编码器是否有反应 |
| `pwm` 为正但 `actual` 为负 | 编码器方向反了 | 交换 A/B 相或在软件中取反速度 |
| `actual` 波动很大 | 采样周期、编码器计数、机械抖动、滤波不足 | 固定 10ms 周期，记录 `delta`，必要时做简单滤波 |
| `run=0` 后电机仍转 | 停止逻辑或 TB6612 控制脚未正确关闭 | 检查停机分支、`Motor_SetPWM(0)` 和方向脚状态 |
| 命令输入无效 | 串口中断、命令拼接、换行符处理 | 先确认 USART1 IRQ，再打印接收到的命令字符串 |
| `kp=0.5` 后打印参数没有变化 | 命令解析或 `PID_SetGains()` 接口问题 | 检查 `Uart_HandleCmd()` 到 `Motor_SetKp()` 再到 `PID_SetGains()` 的调用链 |
| `%f` 打印异常 | newlib-nano 默认不支持浮点 printf | 改成参数乘 1000 后按整数打印，或开启 `_printf_float` |
| 下载后串口无输出 | 波特率、TX/RX 接线、USART 初始化、串口助手设置 | 检查 115200-8-N-1、PA9/PA10、`MX_USART1_UART_Init()` 和 `Uart_Init()` |
| 一 `run=1` 就猛转并故障 | 编码器方向反、A/B 输入浮空、目标源不是预期值、故障后状态掩盖触发瞬间 | 先固定 `source:Uart` 和 `target`，再看 `PWM` 与 `actual` 正负关系，最后查 A/B 上拉和走线 |
| 拔掉 A/B 其中一根线反而正常 | 双相信号解码异常、某一路浮空或受干扰 | 两相都接回，PA15/PB3 改上拉；必要时外接 10k 上拉并检查 A/B 方波 |
| 电机一卡一卡、震动但不是快速转动 | 负 PWM 在单向目标下形成反向刹车 | 看 `actual > target` 时 `PWM` 是否变成负数；当前若只做单向调速，可先禁止正目标下的负 PWM |
| `kp` 稍微增大就卡顿或超速 | Kp 偏大、目标过高、测速量化、负 PWM 制动 | 固定目标源，记录 `target/actual/delta/PWM/fault`，从较小 Kp 重新测，不要先怀疑程序改坏 |
| PWM 多数为正但电机仍抖 | PWM 落入电机死区、纯 P 没有基础驱动力 | 看接近目标时 PWM 是否降得太低；负 PWM 限制和死区补偿要分开处理 |
| 固定 PWM 有时无法从静止启动 | 静摩擦、齿轮啮合位置、启动滞回 | 区分启动 PWM 和维持 PWM；用启动助推打破静摩擦，再切回 `PWM_ff + PID` |

## 正式问题记录

### 问题 1：串口有状态输出，但发送 `run=1` / `help` 没反应

| 记录项 | 内容 |
|---|---|
| 问题 | 串口助手能看到周期状态输出，但发送 `run=1` 后 `enable` 仍为 0，发送 `help` 也没有帮助信息返回 |
| 观察 | 串口输出类似 `ms:60006,enable:0,State:IDLE,source:ADC,target:503,actual:0,delta:0,PWM:0,adc:2026,fault:NONE,kp:0.50,ki:0.10,kd:0.10` |
| 判断 | 这类状态输出可能来自 `main.c` 里的 10 秒周期自动打印，只能证明 STM32 的 TX 能发数据，不能证明 STM32 已经收到并解析了命令 |
| 处理 | 检查 `Uart.c` 的接收逻辑：当前只有收到 `\r` 或 `\n` 才认为一条命令结束，然后才会设置 `s_cmd_ready=1` 并交给 `Uart_Task()` 解析。因此串口助手发送命令时需要勾选追加换行，建议使用 `CRLF`，也就是发送 `run=1\r\n`、`help\r\n` |
| 结果 | 勾选换行后再测试；如果仍无响应，再检查 USB-TTL 的 TX 是否接到 STM32 PA10、GND 是否共地、波特率是否为 115200 |
| 复盘 | 看到周期状态输出时，不要直接认为“串口收发都正常”。要区分自动上报和命令响应：发 `help\r\n` 后如果立刻返回帮助信息，才说明 RX、换行结束、命令解析、`Uart_Task()` 全链路正常 |

### 问题 2：电机能转但 `actual/delta` 一直为 0

| 记录项 | 内容 |
|---|---|
| 问题 | `run=1` 后电机正常转动，但 `actual=0, delta=0`，随后 PID 输出逐渐增大 |
| 观察 | 串口示例：`target:50,actual:0,delta:0,PWM:73`，之后 `PWM:123`、`PWM:223`、`PWM:773`，但 `actual/delta` 仍为 0 |
| 判断 | 电机驱动链路正常，问题在编码器反馈链路。PID 看到实际速度一直为 0，会继续增大输出，最后可能触发 `PWM_SATURATION` 或 `ENCODER_LOST` |
| 处理 | 检查编码器 A/B/VCC/GND 接线，确认 A/B 接到 TIM2_CH1/TIM2_CH2 对应引脚；确认编码器和 STM32、驱动板共地 |
| 结果 | 修复接线后，电机转动时 `delta` 和 `actual` 开始变化 |
| 复盘 | `PWM` 有变化但 `actual/delta` 没变化时，优先查编码器接线和供电，不要先调 PID |

### 问题 3：编码器 A/B 相方向反导致越调越错

| 记录项 | 内容 |
|---|---|
| 问题 | 接线修复后电机能反馈速度，但运行时出现猛转、饱和或超速 |
| 观察 | 典型现象是正向目标下，电机正转时 `actual` 为负；或者 `PWM` 越加越大，最后 `fault:PWM_SATURATION` / `fault:SPEED_OVER_LIMIT` |
| 判断 | 闭环反馈方向反了。PID 期望正速度，但编码器报告负速度，导致 `error = target - actual` 变得更大，控制器会继续加大错误方向的 PWM |
| 处理 | 交换编码器 A/B 相，或在软件中对 `Encoder_GetDelta()` / `Encoder_Getspeed()` 取反。调试时以“`PWM > 0` 时 `actual` 应主要为正”为判断标准 |
| 结果 | 调整方向后，出现 `actual > target` 时 `PWM` 能变为负值进行减速，说明反馈方向正确 |
| 复盘 | 编码器方向错比没有反馈更危险，因为控制器会主动往错误方向加力 |

### 问题 4：低分辨率编码器在 10ms 测速下 `actual` 跳变

| 记录项 | 内容 |
|---|---|
| 问题 | `actual` 在 `0/214/428...` 之间跳变，看起来像速度忽高忽低 |
| 观察 | 串口示例：10ms 下 `delta=1` 时 `actual=214`，下一拍 `delta=0` 时 `actual=0` |
| 判断 | GA12-N20 当前资料按输出轴 7 脉冲估算，TIM2 `TI12` 模式约 x4 解码，输出轴一圈约 28 count。10ms 内低速时可能只有 0 或 1 个 count，所以速度量化很粗 |
| 处理 | 将速度单位明确为 RPM：`rpm = delta / dt * 60 / 28`；测试低速时可用 50ms 窗口或后续加入速度滤波 |
| 结果 | 50ms 下 `delta=3` 对应约 `actual=128 RPM`，显示更平滑，但控制响应变慢 |
| 复盘 | `actual` 跳变不一定代表电机真实速度突变，可能只是采样窗口太短和编码器分辨率太低 |

### 问题 5：A/B 两相都接入后启动即超速，改上拉后恢复

| 记录项 | 内容 |
|---|---|
| 问题 | TIM2 改为 `Encoder Mode TI1 and TI2` 后，两相同时接入时一 `run=1` 就 `SPEED_OVER_LIMIT`，拔掉 A/B 其中一根线反而能跑 |
| 观察 | 异常示例：`enable:0,State:FAULT,source:Uart,target:300,actual:0,delta:0,PWM:0,fault:SPEED_OVER_LIMIT,kp:0.80,ki:0.00,kd:0.00`；故障后 `actual/PWM` 已被清零，无法直接看到触发瞬间 |
| 判断 | 不是单纯 PID 参数问题。两相同时参与解码时，某一路输入浮空或受电机 PWM 干扰会产生异常计数；PA15/PB3 原来为 `GPIO_NOPULL`，对编码器输入不够稳 |
| 处理 | 在 CubeMX 中将 TIM2_CH1 PA15、TIM2_CH2 PB3 都配置为 `Pull-up`，重新生成、编译、烧录 |
| 结果 | PA15/PB3 上拉后，两相同时接入可以正常运行 |
| 复盘 | 编码器 A/B 是数字输入，不能只看“线接上了”。无上拉、线长、与电机电源线并行、共地不好，都可能让 TIM2 看到假脉冲 |

### 问题 6：速度略超目标后出现震动，限制负 PWM 后恢复

| 记录项 | 内容 |
|---|---|
| 问题 | 编码器和上拉问题解决后，`kp` 稍大时电机不是快速转动，而是表现为一卡一卡、震动 |
| 观察 | 典型现象是 `actual` 略大于 `target` 时，串口出现负 PWM，例如：`target:300,actual:471,delta:11,PWM:-342` |
| 判断 | 在当前单向目标场景下，负 PWM 不是简单减小油门，而是给电机反向制动。低分辨率测速下 `actual` 只要略微超过目标，就可能立刻触发反向拉扯 |
| 处理 | 在 `Control_Update()` 中加入限制：当 `target_speed > 0` 且 `PWM < 0` 时，把 `PWM` 修正为 0，不再允许正向目标下输出负 PWM |
| 结果 | 加上限制后，用户实测 `kp=0.8` 甚至 `kp=1.7` 都可以稳定运行 |
| 复盘 | 先弄清当前场景是不是单向调速。若目标只会是正数，允许负 PWM 往往是在“减速”和“反向刹车”之间选了后者，容易把系统弄成震动 |

### 问题 7：把 Kp 过大误判成程序异常

| 记录项 | 内容 |
|---|---|
| 问题 | 调参时出现 `kp=0.40` 不动、`kp=0.50` 一卡一卡、`kp=0.70` 直接 `SPEED_OVER_LIMIT`，容易误以为程序或硬件又出问题 |
| 观察 | 真实数据：`kp=0.40` 时 `target:552,actual:0,delta:0,PWM:220,fault:NONE`；`kp=0.50` 时 `target:551,actual:685,delta:3,PWM:-67,fault:NONE`；`kp=0.70` 时故障后状态为 `enable:0,State:FAULT,target:551,actual:0,delta:0,PWM:0,fault:SPEED_OVER_LIMIT` |
| 判断 | 这不是典型程序异常。`kp=0.40` 输出不足以可靠启动；`kp=0.50` 已经能让速度超过目标，随后负 PWM 产生制动；`kp=0.70` 启动冲击更大，触发超速保护。故障后状态已被清零，不能用 `actual=0/PWM=0` 反推触发瞬间没有超速 |
| 处理 | 固定目标源和目标值后重新调参；先用 `ki=0,kd=0` 找 P 控制可用范围；Kp 增大后若出现卡顿或故障，先判断是否已经进入振荡/超速，而不是继续改程序 |
| 结果 | 当前结论是 Kp 过大和测速量化共同造成不稳定，代码主链路不作为首要怀疑对象 |
| 复盘 | 调参时必须记录真实串口数据。尤其故障类问题要注意：故障后的 `status` 往往是保护逻辑处理后的状态，不等于故障触发瞬间的状态 |

### 问题 8：PWM 没有明显为负但电机仍然抖动

| 记录项 | 内容 |
|---|---|
| 问题 | `kp=0.47` 时，一段时间内 PWM 多数为正，但电机仍然抖动；偶尔会冒出很小的负 PWM |
| 观察 | 真实数据：`target:552,actual:428,delta:0,PWM:58`；`target:551,actual:85,delta:1,PWM:219`；`target:552,actual:214,delta:-1,PWM:159`；`target:552,actual:557,delta:2,PWM:-2` |
| 判断 | 这类抖动不完全是“负 PWM 反拉”。纯 P 控制下，actual 接近 target 时 error 变小，PWM 会降到很低，例如 `PWM=58`，此时可能不足以克服电机静摩擦、齿轮阻力或负载；速度掉下来后 error 又变大，PWM 跳高，于是形成“推一下、松一下”的循环 |
| 处理 | 将问题拆成两类处理：正向目标下的负 PWM 需要限制为 0，防止反向制动；PWM 太小顶不动需要通过死区补偿、基础驱动力、目标斜坡或少量积分解决 |
| 结果 | 当前判断为控制策略问题，不是串口、编码器或 PID 计算公式异常 |
| 复盘 | 不要把所有抖动都归因于 Kp 过大。PWM 为正但数值太小，也可能等价于“没有足够驱动力”。纯 P 直接当电机驱动力使用时，接近目标就会输出趋近 0，而真实电机维持转速需要非零 PWM |

### 问题 9：前馈 PWM 能维持转动，但不能可靠从静止启动

| 记录项 | 内容 |
|---|---|
| 问题 | 开环测试时，按稳定运行点计算出的 `PWM_ff` 不能从静止可靠启动电机 |
| 观察 | 使用开环数据 `(actual=1328, PWM=220)` 和 `(actual=1242, PWM=210)` 估算 `kS≈66, kV≈0.12`，则 `target=551` 时 `PWM_ff≈132`。实际测试中 `PWM=132/150/170` 可能从静止不转，但手动给电机一点外力后又可以持续转动 |
| 判断 | `PWM_ff = kS + kV * target` 描述的是稳定运行时的维持力，不等于从静止打破静摩擦的力。电机启动还受静摩擦、齿轮啮合位置、转子位置、电源瞬态等影响 |
| 处理 | 增加启动助推状态：`run=1` 时置位 `s_pwm_start_boost_active` 并清零 tick；启动阶段给较大的 `APP_PWM_START_BOOST`；检测到 `encoder_delta != 0` 或助推超时后清除 active/tick，切回正常 `PWM_ff + PID` |
| 结果 | 当前控制策略拆分为启动助推、前馈维持、PID 修正三部分，避免为了启动而把 Kp 或维持前馈调得过大 |
| 复盘 | 算 `kS/kV` 时不要用启动临界点推断启动能力。开环拟合点只说明维持某个速度大概需要多少 PWM；从静止启动需要单独的启动助推或启动斜坡处理 |

## 记录原则

- 观察要尽量来自串口输出、示波器、万用表、实际电机现象，不要只写“感觉不对”。
- 判断要说明为什么怀疑这个方向。
- 处理要写具体动作，例如“交换编码器 A/B 相”或“检查 TIM1_CH1 PWM 输出”。
- 结果要写是否解决，没解决也要记录下一步。
