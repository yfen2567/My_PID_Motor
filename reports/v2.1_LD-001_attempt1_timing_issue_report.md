# LD-001 第一次负载扰动尝试归档（时机问题）

> 本报告归档第一次测试尝试。由于人工扰动时机问题，本次不作为正式 LD-001 结果；日志中的停速异常和原分析均予以保留，不改写为通过。

## 1. 结论摘要

**测试判定：未通过。**

日志能够证明：扰动期间 `actual` 总体下降、PWM 总体提高；但在 `RELEASE_LOAD` 阶段，`actual` 最终降为 0，`delta` 为 0，PWM 升至 140，并持续到 stop 前仍未恢复。因此不满足“释放负载后恢复到扰动前附近”的通过判据。

**根因判定：证据不足。** 原始日志没有记录人工负载的实际大小、实际释放时刻和电机机械现象，不能仅凭串口数据判断是负载未完全释放、电机实际停转、编码器反馈异常，还是其他硬件条件造成。

## 2. 测试目标

在 UART 目标源、`target=500`、`Kp=0.05, Ki=0, Kd=0` 条件下，观察轻微外部负载扰动时：

- actual 是否下降；
- PWM 是否相对扰动前基线提高；
- 释放负载后 actual 与 PWM 是否恢复；
- 全程是否出现 fault；
- stop 后是否进入安全软件状态。

## 8. RELEASE_LOAD 后字段变化

`RELEASE_LOAD` 标记后的最初记录仍为 actual 428～342、PWM 118～122。到 `ms:12984`，actual 变为 0、delta 变为 0、PWM 升至 140。此后至 stop 前最后一条运行状态 `ms:15138`，连续 18 条 RELEASE_LOAD status 均为 `actual:0、delta:0、PWM:140`。

```text

ms:12760,...,target:500,actual:342,delta:1,PWM:122,...,fault:NONE,...
ms:12984,...,target:500,actual:0,delta:0,PWM:140,...,fault:NONE,...
ms:15138,...,target:500,actual:0,delta:0,PWM:140,...,fault:NONE,...
```

从第一条 `actual=0` 到 stop 前最后一条运行状态，固件 `ms` 跨度约 2.15 s。日志没有出现恢复到 500 附近的证据，所以 RELEASE_LOAD 阶段不满足计划中的恢复判据。

## 9. stop 后安全状态

发送 stop 后，同一接收块先出现了一条尚未执行 stop 的运行状态，并随后返回 `OK:CMD_QUEUED`。该中间状态保留如下：

```text

OK:CMD_QUEUED
```

后续 status 显示 stop 已被执行：

```text
ms:15562,enable:0,State:IDLE,source:Uart,target:500,actual:0,delta:0,PWM:0,...,fault:NONE,...
ms:15599,enable:0,State:IDLE,source:Uart,target:500,actual:0,delta:0,PWM:0,...,fault:NONE,...
```

因此，本次日志能够确认最终 stop 后的软件安全状态为 `enable:0、State:IDLE、actual:0、delta:0、PWM:0`。它不能代替 PA8、方向脚或电机电流的物理测量。

## 10. get fault 结果

stop 后发送 `get fault`，返回：

```text
ms:15774,enable:0,State:IDLE,source:Uart,target:500,actual:0,delta:0,PWM:0,...,fault:NONE,...
FAULT_SNAPSHOT_NONE
```

这只能证明本次查询时没有保存的 fault snapshot。它不能把 RELEASE_LOAD 阶段的停速现象解释为“系统无异常”，也不能证明有快照分支可用。

## 11. actual / PWM / fault 分析

1. 扰动前 actual 有非零速度反馈，PWM 稳定在 114～118，形成了可用基线。
2. APPLY_LOAD 阶段 actual 总体下降，PWM 总体提高，软件补偿方向正确。
3. RELEASE_LOAD 阶段没有恢复；实际记录反而发展为 `actual=0、delta=0、PWM=140` 并持续到 stop。
4. `PWM=140` 是软件输出命令，只能说明控制器提高了命令值，不能证明 PA8 实际波形、驱动电流或电机转矩正常。
5. 全部运行阶段均为 `fault:NONE`。尤其是 actual/delta 长时间为 0 时仍未产生 fault，这一现象必须保留；仅凭日志无法判断它是否符合当前故障检测条件。
6. `FAULT_SNAPSHOT_NONE` 与 `fault:NONE` 一致，但不能解释停速根因。

## 13. 证据不足的地方

- MARK 是脚本写入的阶段提示，不能证明人工负载恰好在该时刻施加或完全释放；
- 没有记录负载大小、持续力矩或实际操作方式；
- 没有人工现象记录，无法确认 `actual=0` 时电机是否物理停转；
- 没有 PA8 PWM、方向脚、编码器 A/B 相、驱动电流和电源电压测量；
- 没有固件 commit、构建产物及其 hash，无法把日志绑定到唯一固件；
- 周期 status 与主动查询交织，部分时间点相近，分段均值不适合计算精确恢复时间或动态性能；
- 日志不足以区分负载未释放、机械阻力、供电/驱动能力、物理 PWM、编码器反馈或控制策略中的具体原因。

## 14. 下一步建议

1. 先由测试人员补充现场事实：`RELEASE_LOAD` 提示出现后是否确实完全释放负载，以及 `actual=0` 时电机是否肉眼确认停转。
2. 如果确认负载已经完全释放且电机实际停转，则保留本次“LD-001 未通过”结论，不需要把日志改写成通过，也不应立即根据单次日志修改源码。
3. 如果人工动作时机无法确认，本次仍保留为未通过/根因证据不足；如确需重测，只重测 LD-001，并让 MARK 由实际动作触发或同步记录人工操作，而不是先继续扩大到其他测试。
4. 在 LD-001 现象归类前，不用 `fault:NONE` 或 `FAULT_SNAPSHOT_NONE` 否定已经记录的停速异常。
